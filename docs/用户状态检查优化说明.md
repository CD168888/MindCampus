# 用户状态检查优化说明

## 📝 优化背景

在问卷发送功能中，之前只检查了是否存在重复记录，但没有检查用户的状态。这可能导致向**被禁用的用户**发送问卷，浪费资源且不符合业务逻辑。

## ✅ 优化内容

### 1. 添加用户状态检查

在发送问卷前，增加以下检查流程：

```
学生ID → 查询学生信息 → 获取用户ID → 查询用户信息 → 检查用户状态
```

### 2. 检查逻辑

**文件**：`mc-project/src/main/java/com/mc/questionnaire/service/impl/QuestionnaireServiceImpl.java`

```java
@Override
public boolean sendQuestionnaire(Long questionnaireId, Long studentId) {
    // 1. 根据学生ID查询学生信息，获取用户ID
    Student student = studentInfoMapper.selectById(studentId);
    if (student == null || student.getUserId() == null) {
        return false; // 学生信息不存在或未关联用户
    }
    
    // 2. 根据用户ID查询用户信息，检查用户状态
    SysUser user = sysUserMapper.selectUserById(student.getUserId());
    if (user == null) {
        return false; // 用户不存在
    }
    
    // 3. 检查用户是否被禁用（状态为1表示禁用）
    if (UserConstants.USER_DISABLE.equals(user.getStatus())) {
        return false; // 用户已禁用，不发送问卷
    }
    
    // 4. 检查是否已存在该学生的该问卷测评记录
    EvaluationResult query = new EvaluationResult();
    query.setStudentId(studentId);
    query.setQuestionnaireId(questionnaireId);
    List<EvaluationResult> existingResults = 
        evaluationResultMapper.selectEvaluationResultList(query);
    
    if (existingResults != null && !existingResults.isEmpty()) {
        return false; // 已存在记录，跳过
    }
    
    // 5. 插入新的测评记录
    evaluationResultMapper.insertEvaluationResult(new EvaluationResult() {{
        setQuestionnaireId(questionnaireId);
        setStudentId(studentId);
        setCompletionStatus("0"); // 0-未完成
        setReadStatus("0"); // 0-未读
        setAiStatus("0"); // 0-未分析
    }});
    
    return true; // 发送成功
}
```

### 3. 用户状态常量

使用 `UserConstants` 中定义的状态常量：

```java
// 用户正常状态
UserConstants.NORMAL = "0"

// 用户禁用状态
UserConstants.USER_DISABLE = "1"
```

### 4. 前端提示优化

更新了提示信息，让用户知道跳过的原因可能包括用户被禁用：

**全部跳过时**：
```
该部门的 2 名学生无法发送问卷（可能已发送过或用户被禁用）
```

**部分跳过时**：

## 💡 注意事项

1. **用户状态值**
   - `"0"` = 正常（可发送）
   - `"1"` = 禁用（不发送）
   - `null` 或其他值 = 视为正常（兼容性考虑）

2. **性能影响**
   - 每次发送前需额外查询2次数据库（学生信息 + 用户信息）
   - 对于大批量发送，建议优化为批量查询

3. **跳过统计**
   - 跳过原因包括：已存在记录、用户被禁用、学生信息不存在、用户不存在
   - 统一计入 `skippedCount`，前端提示时说明可能的原因
